TCP/IP의 계층

클라이언트와 서버 모두 크게 4가지 계층을 가지고 통신을 합니다.

이 통신이라는 것이 현실에서의 우편과 상당히 유사한 개념이기 때문에 우편을 가지고 쉽게 적어보려 합니다.

4계층 명칭   우편 설명
App 애플리케이션: HTTP 리퀘스트 편지지 작성 데이터를 어떻게 주고 받을지에 대한 약속 (요청/응답)
OS
트랜스포트: TCP 조각 / PORT 편지지 포장
인터넷: IP   발송/수신자 주소 첨부
LAN 링크: 네트워크 MAC주소 추가 편지 봉투 포장  

서버에 데이터를 어떤 형식으로 요청하는 지는 알아두어야 하기 때문에 HTTP는 필수적으로 알고 있어야 합니다.

HTTP

Request & Response

Request

  • 요청
  • 클라이언트
POST /form/entry HTTP/1.1
Host: www.naver.com
Connection: Keep-alive
Content-Type: application/x-www-form-urlencoded
Content-Length: 16

name=gildong&age=20
  1. POST /form/entry HTTP/1.1: 메소드 + 요청대상(경로) + HTTP버전
  2. Host ~ Content-Length: 부가 정보 (메타데이터)
  3. name=gildong&age=20: 실제 전송할 데이터 (body)

    Response

    • 응답
    • 서버
HTTP/1.1 200 OK
Date: Tue, 10 Jul 2020 06:40:14 GMT
Content-Length: 352
Content-Type: text/html

<html>
...
</html>
  1. HTTP/1.1 200 OK: HTTP버전 + 상태코드 + 문구
  2. Date ~ Content-Type: 부가 정보 (메타데이터)
  3. <html></html>: 실제 전송할 데이터 (body)

요청 메소드의 종류

메소드 설명 제공 HTTP 버전
⭐ GET 리소스 휙득 (조회) 1.0 1.1
⭐ POST 엔티티 (등록) 1.0 1.1
⭐ PUT 파일 전송 (데이터 대체/없다면 생성) 1.0 1.1
⭐ PATCH 리소스 부분 변경 (부분 변경) 1.1
HEAD 메세지 헤더 취득 1.0 1.1
⭐ DELETE 파일 삭제 (삭제) 1.0 1.1
OPTIONS 서포트 메소드 문의 1.1
TRACE 경로 조사 1.1
CONNECT 프록시 터널링 요구 1.1
LINK 리소스 간 링크 관계 확립 1.0
UNLINK 링크 관계 삭제 1.0

응답 메시지의 종류

상태 코드 클래스 설명
1xx Informational 리퀘스트를 받아들여 처리중
⭐ 2xx Success 리퀘스트를 정상적으로 처리
3xx Redirection 리퀘스트 완료를 위한 추가 동작 필요
⭐ 4xx Client Error 서버에서 리퀘스트 이해 불가능 (클라이언트 에러)
⭐ 5xx Server Error 서버가 리퀘스트 처리 실패 (서버 에러)

2xx (Success)

  • 200 OK
  • 201 Created
  • 202 Accepted: 요청이 접수되었으나 처리가 완료되지는 않음
  • 204 No Content: 요청이 성공적으로 수행 되었으나, 응답으로 보낼 메시지가 없음

4xx (Client Error)

  • 400 Bad Request: 요청 구문/파라미터, 메시지 등의 오류
  • 401 Unauthorized: 인증이 되지 않았음을 알려주는 오류
  • 403 Forbidden: 요청이 접수되었으나 처리가 완료되지 않음 (ex: 어드민 권한)
  • 404 Not Found: 요청 리소스가 서버에 없음

5xx (Server Error)

  • 500 Internal Server Error: 서버 내부 문제로 오류
  • 503 Service Unavailable: 서비스 이용 불가 (과부하/점검)

REST API

API라는 것은 단순하게 요청 방식에 대한 약속을 정해둔 것입니다.

하지만 https://(ADDRESS)/1이나 https://(ADDRESS)/2와 같이 요청을 넣어버리면 개발자들 간의 소통이 어려워지는 문제점이 있습니다.

REST API는 이런 부분을 해소하고자 제안된 방안입니다.

RESTFUL API

누가 보더라도 어떤 요청인지 이해가 되도록!
